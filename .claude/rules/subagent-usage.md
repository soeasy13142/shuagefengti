# Subagent / Workflow 使用规范

## 核心原则

- **能不阻塞主会话 → 默认派后台 subagent**
- **只需一句话 / 一眼确认的改动 → 留在主会话**

## 建议用 subagent（`Agent` 工具）

| 场景 | 原因 |
|---|---|
| 新功能纯逻辑开发（utils + 测试） | 独立模块，返回摘要即可 |
| 大规模批量化重构 | 改动量大但规律明确 |
| 全库搜索 + 批量替换 | 输出量大，后台跑完汇报 |
| 跑全量测试 | 输出长，不阻塞主会话 |
| 代码审查效果 | 独立审查 |
| 查文档 / API | 独立信息获取 |
| 多工具并行探索代码库 | 各自独立搜索，汇总结果 |

## 建议用 workflow（`Workflow` 工具）

| 场景 | 原因 |
|---|---|
| 全量代码审查·多维度并行 | 多个维度各一个 agent，汇总出报告 |
| 新功能全链路自动化 | 多阶段有先后依赖，脚本编排 |
| 大范围多文件并行改动 + 工作树隔离 | 各干各的，互不干扰 |
| 深度多源交叉验证研究 | 多 agent 独立查源，汇总交叉验证 |

## 不要用 subagent / workflow

| 场景 | 理由 |
|---|---|
| 单文件 ≤10 行改动 | agent 启动开销 > 改文件本身 |
| typo / 文案小修 | 直接改 |
| 需要频繁来回确认 | subagent 不能 `AskUserQuestion` |
| CSS/UI 微调（需看效果） | 需要迭代反馈 |
| 刚说完立刻跟进的连续小修改 | 不必要打断工作流 |
| 需要理解完整意图才能决策 | 主会话才有全部上下文 |

## 默认行为

1. Subagent 默认后台执行
2. 用户可要求"用 subagent 做 X"或"在前台做"
3. Subagent 完成结果有问题可派新的去修正，无需重跑整个任务

## 本项目适用说明

| 任务类型 | 走 subagent？ | 原因 |
|---|---|---|
| 写 plans（`docs/plans/`） | ❌ 主会话 | 需反复讨论确认 |
| 画 UI（WXML/WXSS） | ❌ 主会话 | 需视觉迭代 |
| 并行写多个独立 utils + 测试 | ✅ 各一个 subagent | 互不依赖 |
| 写工具全部文件（utils→page→test） | ⚠️ 看情况 | 方案已定→可 subagent |
| 合并分支 + 推送 | ❌ 主会话 | 需逐项确认 |
| 同步 PROJECT_HANDOFF.md | ❌ 主会话 | 需用户确认 |
| 调研新技术 / 查文档 | ✅ subagent | 独立信息获取 |
