# 异步任务与轮询

**全局规则**（所有 scene 共用；scene 文档不要再复述 pending / 轮询细节）。

任何 toolCall 返回 `status: pending` 和 `taskId` 时，停在当前业务步骤（不要把「已发起」当成成功），并提醒用户：需在微信开发者工具内完成授权、扫码或操作确认（交互在开发者工具内，不在聊天中）。

## 何时必须主动轮询

**仅**下列会阻塞后续依赖步骤的派生任务，agent 才必须自行进入轮询：

| 场景 | 说明 |
|------|------|
| `login` | 扫码登录未完成则后续业务不可用 |
| `wechatide auth`（CLI 子命令） | client 授权未完成则后续 `wechatide` 调用不可用 |

顺序：

1. **先提醒用户**去开发者工具完成扫码 / 授权
2. **再自行开启轮询**；不要等用户在对话里反馈「已扫码 / 已确认」后再开始

每 10 秒调用：

```bash
wechatide -c <clientName> polling_task_result --task-id <taskId>
```

状态不再是 `pending` 时立即停止，最多轮询 10 次：

- 仅 `success` 表示原操作成功，可以继续依赖步骤
- `failed`、`cancelled`、`expired` 均不得继续依赖步骤
- 第 10 次仍为 `pending` 时停止本轮轮询，告知用户交互尚未完成，并保留任务上下文供后续恢复

## 其他 pending（不要主动轮询）

除此之外的所有 `pending + taskId`（如 `upload`、`auto_preview`、`project_remove`、云写确认等）：

- **提醒用户**在开发者工具内完成确认即可
- **不要**进入 10 秒轮询循环；可结束本轮等待用户稍后继续
- 写入 `pendingTask`（`taskId`、原工具名、非敏感参数摘要、最后状态）
- `pending` 不是失败；**不得**因未轮询完而重发原写操作

用户回来继续、或准备执行与 `pendingTask` 相同的操作时，再查一次旧 `taskId`（见下方「恢复与重发」）。

## 保留任务上下文

主动轮询超限，或其他 pending 暂不轮询而移交时，保留：

- `taskId`
- 原工具名
- 可用于识别同一操作的非敏感参数摘要（不得记录访问令牌）
- 最后状态与已轮询次数（若有）

根 SKILL 移交时写入 `pendingTask`。

## 恢复与重发

后续准备执行与 `pendingTask` 相同的操作时，必须先用旧 `taskId` 调用 `polling_task_result`：

| 旧任务状态 | 处理 |
|------------|------|
| `pending` | 不得重发；`login` / `auth` 继续主动轮询；其他操作告知仍待确认，可再查一次或等用户完成后再查 |
| `success` | 原操作已经成功，不得重发；直接使用原结果继续 |
| `failed` | 允许重新发起，但仍须遵守原工具的参数、确认和安全规则 |
| `cancelled` / `expired` | 不自动重发；用户仍明确要求时才重新发起 |
| 无法查询 / 状态不明 | 不得重发可能产生副作用的操作，先向用户说明旧任务状态无法确认 |

只有旧任务进入明确的非 `pending` 终态，并且按上表允许时，才真正重发原操作。
