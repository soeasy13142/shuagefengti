## 1. 网络工程方向（A 类）

### A1. TLS/HTTPS 握手动画机

**一句话定位**：以序列图动画演示 TLS 1.3 握手全过程 —— ClientHello → ServerHello → 证书验证 → 密钥交换 → Finished，每步展示密码学原语和报文结构。

**展示的专业素养**：
- TLS 协议分层（Record Layer / Handshake Layer）
- 证书链验证（Root CA → Intermediate → Leaf）
- 密钥交换算法（ECDHE：椭圆曲线 + DH）
- 前向安全性（Forward Secrecy）的本质
- 对称/非对称加密在 TLS 中的分工
- TLS 1.2 vs 1.3 的握手轮数差异
- SNI（Server Name Indication）原理
- HSTS / Certificate Transparency 概念

**MVP 功能范围**：
1. 首页入口卡片：`TLS 动画机` 或 `HTTPS 探秘`
2. 新增页面：`pages/tls-viz/tls-viz`
3. 模式选择：TLS 1.3 握手 / TLS 1.2 握手（对比模式）
4. 序列图核心（复用 TCP 动画机的双竖线布局）：
   - Client 竖线 + 状态标签
   - Server 竖线 + 状态标签
   - 报文箭头（ClientHello / ServerHello / Certificate / Finished 等）
   - 每种报文的不同颜色编码
5. 证书链面板：
   - 展示证书层级树
   - 逐级验证动画（签名校验 → 有效期检查 → CN 匹配）
6. 密钥交换细节面板：
   - ECDHE 参数展示（椭圆曲线名、公钥 hex）
   - 可视化 DH 密钥协商过程
7. 步骤详情面板（复用 TCP 动画机模式）：
   - 当前报文名称和类型
   - 关键字段（TLS 版本、密码套件、随机数、Session ID）
   - 状态变化（Client/Server 状态流转）
   - 教学解释 + 面试提示
8. 知识点卡片（可折叠）：对称/非对称加密分工、前向安全性、证书链信任模型、TLS 1.3 改进

**视觉风格**：沿用 Claude Design 暖奶油画布。TLS 序列图在 TCP 动画机基础上增加「锁」图标动画、证书链树形图。

**与现有代码的关系**：
- 大量复用 TCP 动画机的序列图架构（`_buildArrows`、`_applyStep`、播放控制）
- 可抽取共享的「序列图引擎」utility：`utils/sequence-diagram.js`

**实现排期**：约 5-7 天。

### A2. DNS 解析过程可视化

**一句话定位**：输入域名，逐步展示本地缓存 → hosts 文件 → 递归解析器 → 根域名服务器 → TLD → 权威服务器的完整 DNS 查询链路，每一步显示 DNS 报文内容。

**展示的专业素养**：
- DNS 分层命名空间（根域 → TLD → 权威）
- 递归查询 vs 迭代查询
- DNS 报文格式（Header + Question + Answer + Authority + Additional）
- DNS 记录类型（A/AAAA/CNAME/MX/NS/TXT/SOA）
- TTL 和缓存机制
- DNSSEC 基本概念
- Anycast 在根服务器中的应用
- DNS over HTTPS (DoH) / DNS over TLS (DoT)

**MVP 功能范围**：
1. 首页入口卡片：`DNS 解析器`
2. 新增页面：`pages/dns-viz/dns-viz`
3. 输入域名 → 点击「解析」
4. 树形链路图：
   - 根节点：本地 DNS 缓存 / hosts
   - 第二层：递归解析器
   - 第三层：根域名服务器
   - 第四层：TLD 服务器（.com / .cn 等）
   - 第五层：权威 DNS 服务器
   - 每层展开动画：查询方向箭头（→）和响应箭头（←）
5. 每步查询的报文详情：
   - DNS Header 字段（ID、Flags、QDCOUNT、ANCOUNT 等）
   - Question Section（QNAME、QTYPE、QCLASS）
   - Answer Section（返回的 RR 记录）
6. DNS 记录类型速查表
7. 知识点：DNS 缓存投毒、DNS 隧道、DNSSEC 链

**视觉风格**：树形节点动画，类似思维导图展开，查询路径逐级高亮。

**实现排期**：约 4-5 天。

   - **学习模式**：交互式 HTTP 状态码分类卡片 + 缓存流程图

**视觉风格**：终端风格 monospace 字体 + 语法高亮。状态码用彩色标签网格。

### A4. IP 报文分片与重组可视化

**一句话定位**：输入一个大包数据（payload 大小），设定 MTU 值（如 1500），自动计算分片数量，可视化每片的 IP 头字段（Identification、Flags、Fragment Offset）和重组过程。

**展示的专业素养**：
- IP 报文头结构（20 字节逐字段）
- MTU 和路径 MTU 发现（PMTUD）
- 分片三字段：Identification（标识同一数据报）、Flags（DF/MF）、Fragment Offset
- 分片偏移量（以 8 字节为单位）
- 重组超时和丢片处理
- IPv4 vs IPv6 分片差异（IPv6 中间路由器不分片）
- 为什么 TCP 要避免 IP 分片（MSS 协商）
- UDP 分片的坑

**MVP 功能范围**：
1. 首页入口（可作为子网计算器的扩展，同页面 tab 切换，或独立页面）
2. 输入区：
   - 原始数据报大小（如 4000 字节）
   - MTU 值滑块（576-9000，默认 1500）
   - DF 标志位 checkbox
3. 分片可视化：
   - 每个分片的 IP 头部逐字段展示（Version、IHL、Total Length、Identification、Flags、Fragment Offset、TTL、Protocol、Header Checksum、Source/Dest IP）
   - MF/DF 位高亮
   - Fragment Offset 计算过程展示（分段区间标注）
4. 原始 payload 条形图（一整条 → 切分为多段，每段着色）
5. 重组过程动画（分片乱序到达 → 按 Offset 重组）
6. 知识点卡片：为什么要分片？MSS vs MTU、Path MTU Discovery

**视觉风格**：条形图切分 + IP 头部表格。与子网计算器风格统一。

**与现有代码的关系**：
- 可复用 `utils/subnet.js` 中的二进制转换和位运算函数
- 可作为子网计算器的「分片」子 tab

**实现排期**：约 3-4 天。

### B1. 进程调度算法可视化

**一句话定位**：输入一组进程（到达时间、执行时间、优先级），以甘特图动画形式逐个时间片展示 FCFS、SJF（非抢占/抢占）、优先级调度、Round Robin 四种调度算法的执行过程，实时计算平均等待时间和平均周转时间。

**展示的专业素养**：
- 进程状态转换（就绪 → 运行 → 阻塞 → 终止）
- 调度算法的权衡：吞吐量 vs 响应时间 vs 公平性
- 抢占 vs 非抢占
- 时间片大小对 RR 的影响（上下文切换开销）
- 饥饿（Starvation）和老化（Aging）
- 多级反馈队列（MLFQ）是现代 OS 的实际方案
- CFS（Linux 完全公平调度器）的基本思想

**MVP 功能范围**：
1. 首页入口卡片：`进程调度`
2. 新增页面：`pages/cpu-sched/cpu-sched`
3. 输入区：
   - 进程表格：名称、到达时间、执行时间（burst time）、优先级（可选）
   - 预设示例进程组（经典教材示例）
   - 时间片大小滑块（RR 模式）
4. 算法选择：FCFS / SJF（非抢占）/ SRTF（SJF 抢占）/ Priority（非抢占）/ Priority（抢占）/ Round Robin
5. 甘特图可视化：
   - 横轴时间，每个进程的 CPU 占用段着色
   - 当前时刻竖直游标线
   - 就绪队列实时显示
6. 对比模式：同时展示 2-3 种算法的甘特图
7. 统计面板：
   - 每进程：等待时间 / 周转时间 / 响应时间
   - 全局：平均等待时间 / 平均周转时间 / CPU 利用率
8. 播放控制（复用排序可视化模式）：播放/暂停/步进/速度调节
9. 学习卡片：抢占的必要性、时间片如何选取、多级反馈队列简介

**视觉风格**：Claude Design 暖奶油画布 + 甘特图横条（各进程不同着色）。

- 播放控制可复用/抽取为共享组件

**测试计划**：

**MVP 功能范围（一周版本——只做核心地址转换 + 页面置换）**：

### B3. 死锁检测与银行家算法模拟器

**一句话定位**：交互式构建资源分配图（进程节点 + 资源节点 + 分配边 + 请求边），可视化死锁检测过程（资源分配图化简）和银行家算法的安全性检查。

**展示的专业素养**：
- 死锁四个必要条件（互斥、持有并等待、不可抢占、循环等待）
- 资源分配图（RAG）和等待图（WFG）
- 死锁预防 vs 死锁避免 vs 死锁检测+恢复
- 银行家算法（Banker's Algorithm）的安全性检查
- 安全序列的求解过程
- 实际 OS 如何处理死锁（鸵鸟策略、重启、kill 进程）

**MVP 功能范围**：
1. 首页入口卡片：`死锁模拟`
2. 新增页面：`pages/deadlock/deadlock`
3. 两种模式：
   - **死锁检测模式**
     - 手动添加进程节点和资源节点
     - 绘制分配边（资源→进程）和请求边（进程→资源）
     - 点击「检测」，逐步化简资源分配图
     - 判断：可完全化简 = 无死锁，存在不可化简边 = 死锁
     - 死锁环高亮
   - **银行家算法模式**
     - 输入矩阵：Allocation / Max / Available
     - 输入新请求
     - 逐步演示安全性检查
     - 输出安全序列或拒绝请求
4. 预设示例：哲学家就餐问题（5 进程 5 资源）
5. 知识点卡片：死锁的四个条件、鸵鸟策略

**视觉风格**：资源分配图用节点+有向边，可拖拽节点。死锁环红色高亮闪烁。

**工具模块**：`utils/deadlock.js`
- `detectDeadlock(allocation, request, available)` → { hasDeadlock, cycle }
- `bankerSafetyCheck(allocation, max, available)` → { safe, sequence }
- `bankerRequest(allocation, max, available, request)` → { granted, sequence }

### D4. 抽象语法树（AST）构建器

**一句话定位**：输入一个简单的算术表达式（如 `3 + 4 * (2 - 1)`），逐步展示词法分析 → 语法分析 → AST 构建的完整过程，最终生成可交互的树形图。

**展示的专业素养**：
- Token 流到 AST 的映射
- 运算符优先级和结合性（* 优先于 +，括号提升优先级）
- 中缀表达式到树结构的转换
- AST 节点类型（BinaryOp、UnaryOp、Literal、Identifier）
- 语法树 vs CST（具体语法树）/ 解析树
- AST 在编译器后端（语义分析、代码生成）中的作用
- 手写递归下降解析器 vs Parser Generator

**MVP 功能范围**：
1. 首页入口卡片：`AST 构建器`
2. 新增页面：`pages/ast-builder/ast-builder`
3. 输入区：算术表达式（支持 + - * / ^ 括号，变量/数字）
4. 三步可视化：
   - **Step 1：词法分析**（Token 流，复用 D3 lexer 逻辑）
   - **Step 2：递归下降解析**
     - 展示递归调用栈
     - 当前 Token 指针
     - 每个产生式匹配过程
   - **Step 3：AST 树**
     - 交互式树形图（可展开/折叠节点）
     - 前序/中序/后序遍历（生成不同表达式形式：前缀/中缀/后缀）
     - 表达式求值（代入数值计算 AST 结果）
5. 对比展示：中缀表达式 → 前缀表达式 → 后缀表达式（仅 AST 遍历顺序不同）
6. 知识点卡片：AST vs CST、表达式求值

**工具模块**：`utils/ast-builder.js`
- `tokenize(expression)` → tokens[]
- `parseExpression(tokens)` → AST
- `evaluate(ast, context)` → value
- `toPrefix(ast)` → string
- `toPostfix(ast)` → string

**视觉风格**：树形图节点用圆角矩形，运算符节点和操作数节点不同颜色。

**实现排期**：约 4-5 天（解析器 + 树图渲染）。
